Crafts — specification & live channels

A craft is the single lane for all generated UI — from a confirm card to a live, bidirectional dashboard. The engine authors a craft; the product renders it, stores it, and runs its live logic. Live data flows through channels — an independent, subscribable interface to connected assets. The engine reasons only to author a craft and on rare wakes — never per data tick.

Authoring: the engine Storage & live logic: the product Live data: channels (per asset) Formats: html · md · code · svg · mermaid · react

1What a craft is

A craft is content the engine authors and the product renders. It is the single lane for all generated UI. Interface elements you used to think of as separate widgets — confirm cards, chips, steppers, checklists, tabs, tables, charts, banners, documents, dashboards, calendars, live maps — are all crafts. There is no fixed catalog of widget types; the engine generates each one on demand, predominantly as HTML, and as React when the UI needs to be reactive.

A craft can be static or live. A live craft carries fields that bind to live data from connected assets and emit actions back. The engine authors the whole craft — markup and its live fields — and the product takes it from there:

The engine — authors

Decides what craft to build and writes it once: the markup plus the live fields (what to subscribe to, how to shape data, what actions exist, when to step in). Then it stops. It reasons again only when a craft action or a trigger asks it to. Reasoning tracks decisions, not data.

The product — renders & runs

Stores the craft in its own DB, renders it, subscribes to the channels it names, runs the craft's live logic (shaping & triggers), and asks the engine only for reasoning and asset relay. No engine reasoning per tick.

2The axes

Every craft is described by three axes — these are the whole model.

AxisValuesMeaning
surfaceinline · panelinline = in the conversation flow (confirm card, chips, small table). panel = set apart in a window/canvas (document, dashboard, live map).
formathtml · markdown · code · svg · mermaid · reactWhat the content is. html carries most interface elements. react is for reactive UIs — a sandboxed component fed by live data; it never fetches its own data (the product injects it). markdown for prose, code (+ language), svg/mermaid for graphics.
waittrue · falseDoes the stream pause for the user to act? A confirm is wait:true; a banner or a live dashboard is wait:false. A wait:true craft ends the stream in done.status="awaiting".

html/react on inline or panel is the workhorse: every interface element, inline or set-apart, with no widget catalog.

3How a craft is produced — the craft tag

The engine writes the content directly in its output, wrapped in a <craft> tag. A streaming-safe parser buffers until it sees a complete tag, then emits a craft block. The <craft> tag never reaches the product — the product only ever sees craft blocks (§4).

<craft surface="inline" format="html" wait="true" key="confirm-delete" title="Confirm deletion">
  <p>Delete 200 files in <code>/archive</code>? This cannot be undone.</p>
  <button craft-action="yes, delete them">Delete</button>
  <button craft-action="no, cancel">Cancel</button>
</craft>

waitno (default false)true to pause for user interaction

4The craft block

What the product receives after the engine parses the tag. Same envelope as every block. A live craft's data fields live in the same payload — they are the craft, not a separate object.

5Interactivity — three modes

A craft has exactly three kinds of interactivity. Interactivity is the markup — there is no separate controls array.

Mode 1 Local

craft-local. Filter, sort, tab-switch, slider — the engine writes the JS; it recomputes in the browser, sandboxed. The engine never hears it. 0 reasoning, 0 network.

Mode 2 Action

craft-action="…". A clickable element carries a command. On click the product posts it to /execute with the craft id; the engine reads it as the user's response. A general command ("redraw as a heatmap").

Mode 3 Live

craft-bind binds an element to live data; craft-emit sends an action out, tagged route:direct or route:engine (§6).

<craft surface="panel" format="react" key="dispatch-map" title="Live Dispatch">
  <select craft-local name="status_filter">…</select>           <!-- Mode 1: browser-only -->
  <MapCanvas craft-bind="/active" colorBy="status" />         <!-- Mode 3: data IN -->
  <button craft-emit="reassign"  route="direct">Reassign</button>  <!-- Mode 3: action OUT -->
  <button craft-emit="rebalance" route="engine">Rebalance</button>
  <button craft-action="redraw as a heatmap of delays">Heatmap</button> <!-- Mode 2 -->
</craft>

wait is orthogonal — it only decides whether the stream pauses for a Mode-2/Mode-3 action; it is not a fourth mode. Mode-1 interactions never affect it (they don't reach the engine).

6Routing — direct vs engine

Routing is a reasoning judgment — "does this click need the engine, or is it a mechanical asset call?". The engine knows the semantics of every control it just wrote, so it stamps the route once, at authoring time. The product never classifies — it obeys the stamp.

Both routes reach the same asset through the same asset_directory — only the path through the engine differs.
Safety default

Default to route:engine when uncertain or high blast-radius (bulk ops, irreversible changes). A misrouted direct silently bypasses the engine on something that needed it — so the safe failure is to involve the engine. For an action whose routing depends on runtime data, the engine authors it as route:engine rather than encoding a branch in markup.

7A craft's live data

A live craft carries four kinds of field — all part of the craft, all authored by the engine and run by the product.

FieldWhat it isRuns where
subscribeWhich channels (asset connections) to attach to, the local name (as) each binds to, and the desired cadence (poll_s).product opens the subscription
shapeDeterministic formulas that derive points from raw asset data (avg ETA by zone, stopped_ratio, merge(…)). No reasoning.product, per update
bindWhich UI element consumes which (raw or shaped) data path.product applies to the rendered craft
actionsWhat the user can emit, where each routes (direct/engine), and the guard.product routes; engine or channel executes
triggersConditions on shaped data that should involve the engine (a threshold, an anomaly).product evaluates → calls the engine on fire
React, fed not fetching

A format:"react" craft renders in the same sandbox as HTML (JS on, no network/storage). It never opens its own connection — the product injects the channel's live data into its props/state from the subscription. Reactivity for the UI; the channel stays the single data source.

Derived points must be formulas, not reasoning

shape formulas are deterministic and run by the product for free. Anything needing judgment is not a per-tick formula — it is a route:engine action or a trigger, so the engine runs only when it matters.

8Channels — the interface to assets

A channel is an independent, shared, server-side object — one per asset connection. It owns the poll loop, the latest state, a small delta buffer, and a set of subscribers. It does not know what a craft is; a craft is simply one subscriber. Because credentials live in the engine, every asset interaction goes through a channel.

Why per asset connection

A channel maps 1:1 to an asset_directory connection (a credentialed ref_id), so per-asset policy — auth, rate limit, circuit breaker, health, cadence — sits naturally on it. Many crafts that use the same asset subscribe to the same channel: the asset is polled once and the result is fanned out to every subscriber. 50 dashboards on one fleet asset = one poll loop (~0.5 req/s), not 50.

One asset → one channel → one poll loop, fanned out to every subscribing craft across every thread.

How a craft subscribes

The channel is the shared object; the subscription is a craft attaching to it. One craft opens one live SSE and the engine multiplexes all of its channels onto it — so a 5-asset calendar has one connection, not five.

// 1. The craft names its channels (part of the craft, §4):
"subscribe": [ {"channel":"sap_fsm:refA","as":"trucks","poll_s":2},
               {"channel":"gmaps:refB","as":"places","poll_s":30} ]

// 2. Product opens ONE subscription for the craft (engine is stateless, so it sends the list):
POST /channel/subscribe   { thread_id, craft_id, channels:[…] }   → { session_id }
GET  /channel/{session_id}/stream                          // long-lived SSE

// 3. Engine, per listed channel: lazily CREATE if absent · REGISTER session · ref-count++ ·
//    start poll loop if first subscriber · send current SNAPSHOT immediately.

// 4. The poll loop fans a delta to every subscriber; the session fans them IN, tagged by `as`:
event: data
data: { "channel":"sap_fsm:refA", "as":"trucks", "op":"delta", "rev":5821,
        "payload":[ {"truck_id":"T1","lat":…,"lng":…,"status":"moving"} ] }

// 5. Action out (route:direct): product posts to the channel; relayed to the asset.
POST /channel/{channel_id}/request   { op, args }      // result appears to all subs on next poll

Fan-out + fan-in

One channel → many subscribers (fan-out). One subscription → many channels, multiplexed onto one SSE (fan-in).

Mixed cadences

The channel polls at the tightest requested poll_s once and delivers to all; slower subscribers coalesce. One poll, never N.

Reconnect

On an SSE drop the product reconnects with after=<rev> → the channel replays from its delta buffer, or sends a fresh snapshot if the gap is too large.

Lifecycle — keyed by connection, but lives by subscription

"Per asset connection" describes how a channel is keyed (so crafts can share one poll). It does not mean the channel lives as long as the connection. Three separate things, three separate lifecycles:

ThingWhat it isLifecycle
Asset connectionThe credentialed MCP connection in asset_directory.Persistent infrastructure — created when the user connects the asset, managed by connection_manager + the health/refresh schedulers. Crafts never create or destroy it.
ChannelThe poll loop + fan-out over that connection.Lazily created on first subscription; dropped on last unsubscribe. Keyed by the connection so it can be shared.
SubscriptionA rendered, open craft attached via a live SSE stream.The unit the channel ref-counts. A craft stored in the product DB but not open is not a subscriber.
The rule

A channel's life tracks live subscriptions, never stored crafts. The engine only ever asks "does this channel still have a live subscriber?" — it never tracks craft storage or deletion. A channel polls only while a craft is actually on someone's screen.

SituationWhat happens to the channel
User closes the craft / navigates awayThe SSE stream closes → subscriber removed → ref-count−−. At 0 the poll loop stops; the connection stays.
Craft deleted from the product DBIf open, the product closes its stream → same path. If already closed, no subscriber existed → nothing to do. The engine never tracks deletion — it's a product concern.
Transient drop / page refreshAt ref-count 0 the channel lingers for a short grace window (keeps latest state, pauses polling) so a quick reconnect resumes instantly before full teardown.
Browser crashes (no clean close)The SSE ping heartbeat / write-failure detects the dead stream and reaps the subscriber. No zombies.
No craft uses an assetThe channel simply doesn't exist (lazy). The connection idles in asset_directory, ready for the engine's own route:engine calls or a future subscription. Nothing polls, nothing leaks.

Optional: the engine's existing ActivityManager (idle / away / disconnect timeouts) can downsample or pause a subscription whose tab is idle or backgrounded — so an open-but-unwatched craft stops driving fast polls without fully unsubscribing.

Transport is SSE for data down and HTTP POST for actions up — the same transport family as /execute and as MCP's own Streamable HTTP.

9Updating a craft

Two kinds of update. A data change flows through polling and never re-emits the craft. A craft edit (a new data point or control) is a deliberate re-emit.

Data change — the no-re-emit rule

When route:engine runs, it mutates the asset and stops. It does not re-emit the craft — the craft's polling picks up the change on the next tick, exactly as a route:direct change would.

One source of truth for live data: the asset, via polling. The engine never pushes a competing UI state.

Craft edit — the engine is a stateless editor

The craft lives in the product's DB; the engine stores nothing about it. So an edit — a new data point ("also show fuel level") or a new control ("add a sort-by-ETA button") — works like a stateless edit service: the product sends the current craft to the engine, the engine returns the edited version, the product saves it.

The engine never opens a craft — it edits the one it is handed and returns version+1. No engine-side craft storage, no drift.
POST /execute
{ "thread_id":"thr_8f2",
  "input":{ "kind":"block_input", "block_id":"0197-craft-map",
    "value":{ "edit":"add fuel level per truck",
             "current":{ /* the craft's content + live fields — the product attaches it */ } },
    "text":"add fuel level" } }

Data points edit the live fields; controls edit the markup; both re-emit version+1. The disruption is minimal — live data repopulates from the channel on the next poll, and format:"react" reconciles by key so even local UI state survives.

10Streaming & versioning

While the engine writes a craft, the engine streams the craft block with the same id, content appending — the panel fills in live. A version bump resets and re-streams.

  • craft appends content while a version streams.
  • An edit re-authors the same key: the engine re-emits the same id with version + 1; the product clears and re-streams. For format:"react", keyed reconciliation preserves live data and local state across the bump.
  • Re-emit is reserved exclusively for deliberate edits — never for data changes (§9).

11Gating

GateDefaultControls
craftfalseMay the engine emit crafts at all. false ⇒ content stays inline as plain text; a programmatic caller gets a clean text stream.
craft_waittrueMay a craft wait for input (halt-and-ask). Only meaningful when craft:true. false ⇒ crafts may still be interactive but never freeze the run.

A waiting confirm needs craft:true and craft_wait:true. Engine-enforced permission prompts (destructive-action confirms) are emitted as waiting crafts whenever craft is on. Live crafts require craft on; with it off, a live craft degrades to a plain-text snapshot.

12The product's renderer

Crafts need one renderer — not a catalog of named widgets. It must:

  1. Render html/react in a sandboxed container with JS enabled but no network/storage (so Mode-1 logic and React run); also markdown, code (with language), svg, mermaid (fall back to code).
  2. Place by surface: inline in the flow, panel set apart.
  3. Bridge Mode 2: on [craft-action] click, snapshot [name] inputs and post to /execute with the craft id.
  4. Bridge Mode 3: open the craft's channel subscription; apply incoming data to [craft-bind] targets (running the craft's shape formulas); on [craft-emit] route by the stamped route — direct to the channel, engine to /execute.
  5. Reconcile by id: append content within a version; on a version bump, clear and re-stream (React reconciles by key).
  6. Store the craft in its own DB as the system of record; attach it when requesting an edit.

That single renderer covers confirms, selects, chips, tables, charts, documents, calendars and live maps — they are all just markup the engine wrote.

13Invocation — route:engine

A route:engine action reuses the existing contract exactly — no new mechanism.

  • The thread itself. One thread per thread_id (the Coordinator owns the index: coordinator._get_or_create_thread(thread_id)). A route:engine action resumes that thread — it does not spawn a new one.
  • It's a /execute call with thread_id + block_id — the block_input shape:
POST /execute
{ "thread_id": "thr_8f2",
  "input": { "kind": "block_input", "block_id": "0197-craft-map",
            "value": { "objective": "fuel" }, "text": "rebalance fleet" } }
A thread need not be "awaiting"

In the live flow the authoring turn usually ended done.status=completed and the craft lives on in the product. A thread persists server-side; a fresh /execute on its thread_id continues it whether the last turn was completed or awaiting. The Coordinator restores it via _get_or_create_thread. So route:engine is just the next turn on this thread, triggered by a control instead of typed text. route:direct never touches a thread.

14MCP reality (why the engine is the hub)

Assets feed the engine, not the browser

MCP's transports are stdio and Streamable HTTP — request/response, not a firehose; and MCP is not browser-facing. So assets feed the engine; the product reaches them only through channels. The engine is necessarily the hub (credentials live here).

One poll = one batch

Polling SAP every 2s returns all 500 trucks in one response. "Hundreds of msgs/sec" was never the shape — it's one batched request per cadence, comfortably within request/response.

Maps: data vs rendering

The Google Maps MCP gives data; the map rendering is the Google Maps JS SDK loaded inside the browser craft. The engine feeds data points; the SDK draws the map.

Many assets, one craft

The engine holds many asset connections at once. A craft can subscribe to several (calendar threads five), and the product composes them into one surface.

15Where it runs in the engine

The channel is a new subsystem on EngineState, booted next to asset_directory, sitting between it and server.py, driven by the engine's reasoning.

Build checklist (engine side)

1 · Rename channel_state → resume_state.   2 · New module src/channel/ (Manager + Channel).   3 · Add channel_manager to EngineState; construct after AssetDirectory; .start() + register_cleanup.   4 · Poll loop = refresh_scheduler.py pattern, per channel.   5 · Data in/out via asset_directory.   6 · Add subscribe / stream / request routes to server.py.   7 · Teach the engine to author live crafts; handle route:engine via block_input.

17End-to-end

Author once → subscribe → poll & fan-out → product renders → direct actions skip the engine → triggers/engine actions involve it → asset mutated → polling closes the loop. The engine touches the craft twice in its life, never per tick.

18Open items — to pin down during build

This document is build-ready as architecture. The items below are illustrated by example here but are not yet formally specified — they are deliberate build-time decisions. The first three (the expression languages) are the highest-risk surface: the engine must author them correctly and the product must execute them identically.

Sequencing

Design items 1–3 (the transform / trigger / guard languages) first and deliberately — they are a small DSL the engine authors and the product runs, and everything else composes on top of them. Items 4–6 are mechanical and can follow.
